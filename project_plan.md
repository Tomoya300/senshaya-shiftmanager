# シフト連絡自動化ツール プロジェクトプラン

**Version 1.0  |  2026年4月**

---

## 1. プロジェクト概要

### 1.1 背景

カナダの洗車店において、翌日のシフト連絡を全従業員に対して個別のSMSテキストメッセージで送信している。洗車業務は天候により客足が大きく変動するため、翌日のシフトはその日の終業後に決定される。また、開店時は少人数、昼以降に向けて段階的に人員が増えるシフト構成となっており、各従業員に異なるシフト時間を伝える必要がある。

現在マネージャーは、終業後にスタッフルームのホワイトボードにシフトを記入し、その後一人ずつ手動でテキストメッセージを送信している。この作業を効率化するツールを開発する。

### 1.2 目的

- マネージャーのシフト連絡作業時間を大幅に短縮する
- シフト情報・リクエストオフ・労働時間制限などの情報を一元管理する
- 既存のホワイトボード運用を完全には置き換えず、共存できる形で導入する
- ランニングコスト$0で運用可能な構成を実現する

### 1.3 開発体制

- 開発者2名による共同開発
- 利用者:店舗マネージャー1名、アシスタントマネージャー1名
- 対象従業員数:現在15〜16名(最大20名程度)

---

## 2. 業務上の制約と要件

### 2.1 シフト管理の特性

- シフトは前日の終業後に決定される(週次・月次の一括作成は不可)
- 従業員ごとに異なる出勤時間が設定される(例:9時、10時、11時、13時など)
- 休みの従業員も日々変動する
- マネージャーのシフトは原則9時固定。ただし休みの場合は変動あり
- マネージャー休日時はアシスタントマネージャーが代行管理

### 2.2 従業員側の事情

- ワーキングホリデー・Co-op留学のスタッフが多く、入れ替わりが頻繁
- ビザ種別により週間労働時間に上限がある場合がある
- 曜日固定で出勤不可な従業員もいる(他職場との掛け持ち等)
- 特定日・期間のリクエストオフがある
- 従業員のスマートフォンOSは混在(iOS/Android)

### 2.3 既存ワークフロー

- スタッフルームのホワイトボードを情報のマスターとして使用
- ホワイトボードの構造:左端に従業員名、上端に日付・曜日、各セルにシフト記入
- 名前と日付の間に連絡先・備考欄が存在
- 従業員はホワイトボードを写真撮影してシフトを確認
- マネージャーが個別にテキストメッセージで連絡

## 3. 機能要件

### 3.1 必須機能 (MVP)

#### 3.1.1 認証

- マネージャー・アシスタントマネージャー2アカウントのログイン
- 両者は同等の権限を持つ

#### 3.1.2 従業員管理

- 従業員の追加・編集・削除
- 管理項目:氏名、電話番号、ビザ種別、週間労働時間上限、定期休み(曜日)、備考
- 退職者のアーカイブ機能

#### 3.1.3 シフト入力画面

- ホワイトボード形式のグリッドUI(縦軸:従業員、横軸:日付)
- 各セルへのシフト時間入力(時間選択 or 休み)
- マネージャー固定9時の自動入力
- 曜日固定休みの自動反映
- リクエストオフ期間の自動反映・視覚的表示

#### 3.1.4 リクエストオフ管理

- 従業員ごとに期間指定でリクエストオフ登録
- シフト入力画面でのオフ期間ハイライト表示

#### 3.1.5 シフト連絡送信

- 送信前プレビュー画面(全従業員のメッセージ一覧確認)
- 定型文+シフト時間の変数によるメッセージ生成
- iOSショートカット連携によるボタン1回での個別ループ送信
- 代替手段としてSMSリンク方式 (`sms:` スキーム) のボタン表示
- 送信履歴の保存

### 3.2 Phase 2 機能(MVP安定後)

- シフト承諾チェック機能(マネージャー手動)
- 指定時刻までに未確認の従業員アラート
- 週間労働時間の自動集計とビザ上限との比較表示

### 3.3 Phase 3 機能(将来拡張)

- ホワイトボード写真からの自動シフト読み取り(Claude Vision API使用)
- 従業員からの返信自動取得(Twilio導入時)
- シフト履歴の長期分析・労働時間レポート

## 5. システム構成

### 5.1 全体アーキテクチャ

以下のフローで動作する。

```
[1] マネージャーがブラウザでWebアプリにログイン
[2] 従業員管理画面で従業員情報を初期セットアップ
[3] シフト入力画面で翌日のシフトを入力(ホワイトボード形式)
[4] 入力データはSupabaseのPostgreSQLに保存
[5] 送信プレビュー画面で全員分のメッセージを目視確認
[6] iPhoneでiOSショートカットを起動
[7] ショートカットがWebアプリのAPIエンドポイントから送信データを取得
[8] ショートカット内でループ処理し、各従業員に個別SMSを送信
[9] SMSはマネージャーの通信プラン経由で送信(コスト$0)
```

### 5.2 データモデル(主要テーブル)

| テーブル | 用途・主要カラム |
|---|---|
| `managers` | マネージャーアカウント / `id`, `email`, `name`, `role(manager\|assistant)` |
| `employees` | 従業員情報 / `id`, `name`, `phone`, `visa_type`, `weekly_hour_limit`, `notes`, `is_active` |
| `recurring_days_off` | 曜日固定の休み / `id`, `employee_id`, `day_of_week (0-6)` |
| `requested_days_off` | リクエストオフ / `id`, `employee_id`, `start_date`, `end_date`, `reason` |
| `shifts` | シフト情報 / `id`, `employee_id`, `shift_date`, `start_time`, `is_off`, `status` |
| `message_logs` | 送信履歴 / `id`, `shift_id`, `sent_at`, `message_body` |

### 5.3 主要API設計

- `GET /api/employees` - 従業員一覧取得
- `POST /api/employees` - 従業員追加
- `PATCH /api/employees/:id` - 従業員情報更新
- `GET /api/shifts?date=YYYY-MM-DD` - 指定日のシフト取得
- `POST /api/shifts/bulk` - 複数シフトの一括登録
- `GET /api/shifts/messages?date=YYYY-MM-DD` - ショートカット用送信データ取得
- `POST /api/shifts/messages/sent` - 送信完了の記録

### 6.1 主要画面

#### 6.1.1 ログイン画面

- メールアドレスとパスワードによる認証
- マネージャー・アシスタント2アカウント想定

## 7. iOSショートカット連携

### 7.1 ショートカットの動作

1. Webアプリの送信用APIエンドポイントにリクエスト送信
2. レスポンスのJSONから送信対象リストを取得
3. 各従業員に対してループ処理
4. Send Message アクションで個別SMSを送信(Show When Run = OFF)
5. 全送信完了後、送信完了APIを呼び出してログ記録

### 7.2 APIレスポンス例

```json
{
  "date": "2026-04-25",
  "messages": [
    { "phone": "+1...", "body": "明日 4/25 (土) は 9:00 からです。" },
    { "phone": "+1...", "body": "明日 4/25 (土) はお休みです。" }
  ]
}
```

### 7.3 注意事項

- 複数電話番号を一度に指定するとグループSMSとなり受信者間で返信が共有されてしまう
- 必ずループ内で1人ずつ個別送信する設計とする
- APIには簡易な認証(トークン)を実装し、不正アクセスを防止する
- ショートカットはマネージャー・アシスタント両者のiPhoneにインストールする

## 9. 共同開発フロー

### 9.1 ブランチ戦略

- `main`:本番デプロイ用、直接コミット禁止
- `develop`:開発用統合ブランチ
- `feature/xxx`:機能ごとの作業ブランチ
- 各機能ブランチから`develop`へPR、レビュー後マージ
- `develop`から`main`へは安定版確認後にマージ

### 9.2 コミットルール

- Conventional Commitsに準拠(`feat:`, `fix:`, `refactor:`, `docs:` 等)
- コミットメッセージは英語または日本語で統一
- 1コミット = 1つの論理的変更

### 9.3 レビュープロセス

- PRには変更内容の説明と動作確認手順を記載
- 最低1名のレビュー承認後にマージ
- CIが通っていることを確認

### 9.4 コミュニケーション

- GitHub Issuesでタスク管理
- 週1回程度の進捗確認(対面 or オンライン)
- 仕様の疑問点はIssueまたはPRコメントで議論・記録

## 10. 想定リスクと対策

| リスク | 影響 | 対策 |
|---|---|---|
| シフト入力ミスで誤った内容を全員に送信 | 業務に直接影響、信頼性低下 | 送信前プレビュー画面で目視確認を必須化 |
| グループSMSとして送信されてしまう | プライバシー問題、返信が他者に見える | ショートカット内で必ずループで個別送信 |
| APIエンドポイントへの不正アクセス | 従業員情報の漏洩 | 認証トークンによるAPI保護 |
| マネージャーがホワイトボード更新を忘れる | アプリと現実の不整合 | ホワイトボードを情報源とし続ける運用方針を維持 |
| Supabaseの無料枠超過 | サービス停止 | 使用量モニタリング、超過時はログクリーンアップ |
| マネージャーの離職・引継ぎ | 運用担当者不在 | README・運用マニュアルの整備、アシスタントマネージャーへの共有 |

MVPの運用が安定した後の拡張方向として、以下が考えられる。

### 12.1 機能拡張

- ホワイトボード写真の自動読み取り(Claude Vision API)

MVPの運用実績を踏まえ、同様の課題を抱える小規模店舗(他の洗車店、飲食店、小売店など)向けへの展開も将来的な選択肢となる。ただし本プロジェクトの初期目的はあくまで自店舗の業務改善であり、横展開は別途検討事項とする。
